PRIZE DELIVERY - NEXT STEPS
===========================

Current state:
- Models (PrizeDelivery + PrizeDeliveryLog), enum, and migration are in place.
- create_prize_delivery / create_prize_delivery_log helpers exist.
- ALLOWED_TRANSITIONS map + can_transition() + transition() helper exist
  (transition() checks can_transition, writes a PrizeDeliveryLog, and sets the new status).
- raffles_processor creates the delivery at PENDING_DELIVERY_ADDRESS on a won raffle.
- DONE: winner submits address   (PENDING_DELIVERY_ADDRESS -> PENDING_PICKUP_ADDRESS)
- DONE: creator submits pickup    (PENDING_PICKUP_ADDRESS   -> WAITING_FOR_SHIPMENT)
- Both routes log via transition() and are wired to UI buttons (message + raffle details).

2 of ~5 transitions done. Remaining: shipment -> delivered/accepted, plus the timeout job.

2. BUILD REMAINING TRANSITION ROUTES
------------------------------------
Each route pattern: authz -> validate -> transition() -> commit -> notify.

- [DONE] PENDING_DELIVERY_ADDRESS -> PENDING_PICKUP_ADDRESS
    Winner submits delivery address. Route provide_delivery_address, authz = winner.
- [DONE] PENDING_PICKUP_ADDRESS -> WAITING_FOR_SHIPMENT
    Creator submits pickup address. Route provide_pickup_address, authz = creator,
    PickupAddressForm -> pickup_* columns, pickup_address.html.
- [NEXT] WAITING_FOR_SHIPMENT -> PRIZE_SENT
    Creator marks shipped (optionally set/confirm tracking). Authz = creator.
    Likely a simple POST action (no address form) + a button in the creator branch
    of raffle_details.html gated on status == "waiting_for_shipment".
- PRIZE_SENT -> PRIZE_DELIVERED | DELIVERY_FAILED
    Mark delivered / failed.
- PRIZE_DELIVERED -> PRIZE_ACCEPTED | CONTESTED
    Winner accepts or contests. Authz = winner.
- CONTESTED -> PRIZE_ACCEPTED | PRIZE_REJECTED
    Dispute resolution (likely admin).

5. NOTIFICATIONS ON EACH TRANSITION
-----------------------------------
services/notifications_service is now split into two halves around the commit:
- queue_message* / queue_message_for_raffle / queue_message_for_ticket -> stage the
  in-app Message INSIDE the transaction (call BEFORE commit).
- send_external(user, message) -> email/SMS, call AFTER a successful commit.
Both address routes and the settlement job use this split (external never fires on a
rolled-back transaction).

queue_message_* call so the Message carries prize_delivery_id, then add a matching branch
